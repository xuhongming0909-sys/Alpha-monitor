# Alpha Monitor Constitution

## 核心原则

本文件是本项目的最高级项目约束文档，用于定义项目长期不轻易变化的原则。

除非用户明确同意，否则不得擅自修改本文件中的原则性内容。

本项目最终目的是形成套利网页展示以及推送；
项目更改完成后，发布到云端，无需等待确认；

## 1. 数据真实性
- 必须优先使用真实数据、真实接口、真实文件、真实返回结果。
- 严禁使用假数据、演示数据、占位数据、静态空壳或伪造结果冒充真实结果。
- 如果无法获得真实数据，必须明确说明原因、限制、影响范围和阻塞点。

## 2. 直接执行
- 用户不是程序员。凡是代理可以自行完成的分析、修改、配置、运行、验证，都应直接执行。
- 只有在确实需要用户提供权限、密钥、账号、人工确认或外部资源时，才请求用户介入。

## 3. 职责分离
- 代码必须按职责拆分，避免把抓取、计算、数据处理、数据库、展示、推送混在同一个文件或模块中。
- 展示层不负责数据处理，数据处理层不负责推送，数据库层不混入展示逻辑。

## 4. 隔离与低耦合
- 单一功能的修改不应导致其他功能失效。
- 各功能应拥有相对独立的调用链、参数和边界，避免相互嵌套和隐式耦合。
- 参数设计应清晰、扁平、集中，避免为了改一个参数而联动多个文件。

## 5. 配置集中
- 所有可变参数必须进入 `config.yaml` 或项目约定的统一配置入口。
- 不得将路径、阈值、开关、URL、账号字段等散落硬编码在多个文件中。

## 6. 最小改动与最短链条
- 修改路径应尽量单一清晰，简单需求不应牵动多个职责域一起改。
- 从数据获取到结果展示，应优先采用依赖最短、最容易验证的实现链条。
- 不要为了理论完备性引入不必要的复杂流程或额外抽象。

## 7. 简洁与删除优先
- 代码应简洁有力，以可维护性和可理解性为先。
- 过时代码、重复代码、无效逻辑和历史兼容壳应优先删除，而不是继续叠加。
- 默认追求越改越少、越改越清楚。

## 8. 中文注释
- 核心逻辑必须保留中文注释。
- 注释应说明职责、输入输出、边界条件和关键设计原因。

## 9. 文档驱动变更
- 项目必须维护当前生效的一套计划、需求和技术实现文档，并保持与实际状态一致。
- 任何实质性变更前，应先形成计划并获得用户同意。
- 用户同意后再实施，并同步更新需求文档和技术栈文档，确保文档与实际状态一致。


## 项目结构原则

- `data_fetch/` 只负责抓取与标准化。
- `strategy/` 只负责业务计算与规则判断。
- `presentation/` 只负责页面、接口整形和展示逻辑。
- `notification/` 只负责推送配置、推送格式和推送调度。
- `shared/` 只负责配置、路径、时间、运行态和通用能力。

跨域公共逻辑只能下沉到 `shared/`，禁止插件之间直接乱连。

## 配置原则

- `config.yaml` 是唯一正式配置合同。
- 新增参数时，必须先写入 `config.yaml`，再写代码。
- 敏感值可以来自环境变量，但字段名、用途和回退规则仍必须写在 `config.yaml` 中。

## 开发流程原则

以后统一采用下面这套更简化的流程：

### 第一步：任务开始前先读宪法

每次开始任何任务前，必须先读取并遵守：

1. `CONSTITUTION.md`

### 第二步：先形成 plan

- 每次有项目变更需求，先输出或更新 `refactor_docs/001-monitor-refactor/plan.md`
- 可以直接使用 Codex 的 Plan mode 形成方案
- 在 `plan.md` 未确认前，不进入代码实施阶段

### 第三步：实施前再写 REQUIREMENTS 和 SPEC

如果本轮 `plan.md` 会影响以下任一内容：

- 需求范围
- 页面行为
- 计算口径
- 接口含义
- 配置项
- 部署方式

则必须在改代码前按下面顺序更新：

1. `refactor_docs/001-monitor-refactor/REQUIREMENTS.md`
2. `refactor_docs/001-monitor-refactor/SPEC.md`

完成上述更新后，才能进入实施阶段。

## 实施与验收原则

- 实施后必须做最小验证。
- 至少优先使用现有校验命令，例如：
  - `npm run check`
  - `npm run check:boundaries`
  - `python data_dispatch.py exchange-rate`
  - `python data_dispatch.py ah`
  - `python data_dispatch.py ab`
- 如果修改了计算逻辑，还必须核对是否与 `SPEC.md` 的“业务逻辑与计算字典”一致。

## 治理规则

- 本文件高于临时口头约定。
- 如需修改宪法，必须同时更新：
  - `CONSTITUTION.md`
  - `.specify/memory/constitution.md`
- 宪法例外必须在对应 `plan.md` 中说明原因、影响范围和回收方案。
