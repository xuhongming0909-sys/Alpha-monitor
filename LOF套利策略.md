# LOF套利策略

## 1. 功能定位

`LOF套利` 是一个独立根功能页，与现有 `转债 / AH / AB / 监控 / 分红 / 事件套利 / 可转债抢权配售` 平行存在。

本功能的核心目标不是泛化基金行情展示，而是围绕 `LOF 实时价格` 与 `IOPV` 的偏离，识别可申赎套利、折溢价异常和限购约束下的有效标的，并形成独立页面、独立 API 和独立推送链路。

首期范围固定为：

- 页面
- API
- 独立推送

首期不做人工编辑池，不做自动名称猜测映射，不做假数据补值。

## 2. 页面结构

`LOF套利` 页面采用 `单页主表 + 三个内部筛选` 的结构，不拆成多个根标签。

页面内部固定提供 3 个筛选视图：

- `指数LOF`
- `QDII欧美`
- `QDII亚洲`

默认视图为：

- `QDII欧美`

页面从上到下分为 3 个区域：

1. 顶部监控区
2. 主表区
3. 页面注释区

### 2.1 顶部监控区

顶部区域保留为页面最优先阅读区，但不再展示分散的状态摘要信息，而是聚焦当前最有交易价值的监控结果。

顶部固定展示两个监控区块：

- `限购监控池`
- `非限监控池`

每个监控区块都展示对应列表和核心数据，不额外展示抓取模式、条数统计、更新时间摘要等无关信息。

两个监控池内每条至少展示：

- 代码
- 名称
- 溢价率
- 市场（沪市 or 深市）
- 现价
- 涨跌幅
- 成交额
- 申购状态
- 申购费
- 赎回费
- 时间说明

其中时间说明固定为：

- `QDLL` 返回 `T+3`
- `国内LOF` 返回 `T+2`

### 2.2 主表字段

主表按正式业务口径展示，首期至少展示：

- 代码
- 名称
- 现价
- 涨幅
- 成交额
- 场内份额（万份）
- 场内新增
- 净值
- 净值日期
- 指数涨幅
- 相关指数
- 申购费
- 申购状态
- 赎回费
- 赎回状态
- 托管费
- IOPV
- 溢价率

主表交互固定为：

- 支持排序
- 支持搜索
- 50 行分页
- 支持横向滚动
- 继续复用现有共享表格体系

### 2.3 页面注释区

页面末尾固定增加注释区，用于解释：

- 数据来源
- IOPV 计算逻辑
- 溢价率公式
- 监控入池规则
- 推送规则
- T-1 / T-2 / T+2 / T+3 口径说明
- 风险提示

## 3. 数据来源

首期数据来源固定为你指定的 URL 页面，不做额外自动匹配。

### 3.1 页面来源

固定使用以下 3 类来源：

1. `LOF-指数LOF`：`https://www.jisilu.cn/data/lof/#index`
2. `T+0 QDLL` 欧美市场：`https://www.jisilu.cn/data/qdii/#qdiie`
3. `T+0 QDLL` 亚洲市场：`https://www.jisilu.cn/data/qdii/#qdiia`

这些 URL 中已经包含 `LOF 与相关指数/市场` 的匹配关系，因此首期不额外建立自动匹配逻辑。

### 3.2 抓取优先级

抓取链路固定为：

1. 优先使用 `Firecrawl`
2. `Firecrawl` 失败时降级为直接抓取同一 URL
3. 若仍失败，则返回真实错误，不伪造空壳结果

### 3.3 核心提取字段

首期固定提取以下字段：

- 代码
- 名称
- 现价
- 涨幅
- 成交额
- 场内份额（万份）
- 场内新增
- 净值
- 净值日期
- 指数涨幅
- 相关指数
- 申购费
- 申购状态
- 赎回费
- 赎回状态
- 托管费

### 3.4 来源约束

首期只信任 URL 中已有的真实信息，不做：

- 名称模糊匹配
- 代码自动猜测
- 本地白名单补配
- 手工伪造 IOPV

### 3.5 数据源选择原则

根据 `相关指数` 继续寻找后续计算所需的指数与汇率 API，原则固定为：

- 尽量用少数 API 解决所有指数数据
- 尽量用少数 API 解决所有汇率数据
- 优先选择覆盖面广、稳定、可批量获取的真实接口
- 不为少数样本单独引入大量零散 API
- 若少数样本确实无法覆盖，再补充最小必要的新来源

### 3.6 样本过滤规则

主表仅保留满足以下条件的样本：

- 必须是 `LOF`
- 剔除股票名称带有 `ETF` 字样的样本

## 4. IOPV 估算逻辑

本功能核心是 `IOPV` 与 `现价` 的偏离测算。

### 4.1 基础原则

- 必须使用真实来源字段
- 必须明确区分 `指数LOF / QDII欧美 / QDII亚洲`
- 必须明确区分 `T-1` 与 `T-2` 的净值滞后
- 若字段不足，则返回“无法计算”，不得补假值

### 4.2 溢价率公式

首期统一使用以下公式：

`溢价率 = (现价 / IOPV) - 1`

说明：

- 溢价率为正，表示按该口径存在溢价
- 溢价率为负，表示按该口径存在折价

### 4.3 欧美市场 IOPV 计算

欧美市场数据源为 `T-2` 日净值数据，修正方法固定为：

`IOPV = T-2日净值 x (今日指数点位 / T-2日收盘指数点位) x (今日汇率 / T-2日收盘汇率)`

具体口径如下：

- 今日数据使用实时数据
- 实时数据包含盘前、盘后以及夜盘
- 汇率按指数货币单位兑换人民币计算
- 例如 `1美元 = 7人民币`
- 若某只 LOF 对应多个指数，则按指数权重进行复合计算

### 4.4 指数LOF与亚洲市场 IOPV 计算

`指数LOF` 与 `QDII亚洲` 数据源为 `T-1` 日净值数据，修正方法固定为：

`IOPV = T-1日净值 x (1 + 指数涨幅) x (今日汇率 / T-1日收盘汇率)`

具体口径如下：

- 数据源已经提供当日指数涨幅数据
- 汇率仍按相关指数对应货币换算为人民币
- 若为人民币资产，可视为汇率修正项等于 `1`

### 4.5 多指数情况

若某只 LOF 对应多个指数：

- 仅当 URL 中明确提供权重，或能稳定解析出权重时，才做加权计算
- 若没有明确权重，不做猜测
- 此类样本标记为“组合权重未解析”，可展示但不伪造精确 IOPV

### 4.6 汇率规则

若相关资产涉及外币：

- 必须通过真实汇率换算为人民币口径
- 汇率优先复用现有汇率链路或本轮统一接入的少数汇率 API
- 若币种存在但汇率缺失，则该行 IOPV 标记为“汇率缺失，无法计算”

## 5. 监控规则

本功能监控池采用动态入池和动态出池，不设置人工长期保留池。

### 5.1 限购监控池

同时满足以下条件才进入 `限购监控池`：

1. 申购状态存在限购，且限购金额小于 `10万`
2. 溢价率大于 `1%`
3. 成交额大于 `100万`
4. 动态更新，符合要求的纳入，不符合要求的调出

### 5.2 非限监控池

同时满足以下条件才进入 `非限监控池`：

1. 申购状态不存在限购
2. 溢价率大于 `5%` 或折价率小于 `-5%`
3. 成交额大于 `100万`
4. 动态更新，符合要求的纳入，不符合要求的调出

### 5.3 动态进出规则

- 满足条件即入池
- 不再满足条件自动出池
- 不做人工锁定
- 不做手工新增

## 6. 推送策略

`LOF套利` 的推送必须是独立模块，不并入现有 AH/AB 或其他汇总推送。

### 6.1 推送方式

首期固定为两类推送并存：

- 即时推送
- 定时汇总推送

### 6.2 推送内容

每条推送至少包含：

- 代码
- 名称
- 溢价率
- 市场（沪市 or 深市）
- 涨跌幅
- 成交额
- 申购状态
- 申购费
- 赎回费
- 时间

时间字段口径固定为：

- `QDLL` 返回 `T+3`
- `国内LOF` 返回 `T+2`

### 6.3 即时推送

当某只 LOF 首次纳入任一监控池时：

- 即时推送该 LOF 信息

### 6.4 定时推送

下午固定进行 3 次定时推送，推送两个监控池的全量标的：

- `13:30`
- `14:00`
- `14:30`

## 8. 架构约束

本功能必须严格遵守项目宪法，按职责拆分：

- `data_fetch/lof_arbitrage`：负责抓取与标准化
- `strategy/lof_arbitrage`：负责 IOPV、溢价率和监控判定
- `presentation/`：负责页面、API 整形、展示交互
- `notification/lof_arbitrage`：负责独立推送与调度

不得出现以下情况：

- 展示层直接做计算
- 抓取层直接拼页面 HTML
- 推送层直接重写策略逻辑
- 多个功能共用一份难以理解的 LOF 特例逻辑

## 10. 风险与边界

首期需要明确以下边界：

- 本页面展示的是计算口径下的 `IOPV`，不是交易所官方逐笔 IOPV
- `QDII欧美` 使用 `T-2` 基准净值，滞后更明显
- `指数LOF / QDII亚洲` 使用 `T-1` 基准净值
- 多指数 LOF 若无权重，不能伪造精确值
- 汇率缺失时必须如实标记不可计算
- 页面不能为了完整性而补假字段

## 13. 2026-03-25 盘后净值与监控补充修订

### 13.1 正式溢价率口径

当前正式口径继续使用：

`溢价率 = (现价 / IOPV) - 1`

### 13.2 盘后净值优先

当集思录已经把某只 LOF 的 `净值日期` 更新到当日时：

- 不再继续按 `T-1 / T-2` 路径重复外推
- 直接复用集思录的 `净值溢价率`
- `IOPV` 列回落为同日 `净值` 参考值，确保主表展示与溢价率口径一致

该规则同时适用于：

- `指数LOF`
- `QDII欧美`
- `QDII亚洲`

### 13.3 欧美净值日期对齐

`QDII欧美` 在仍需外推时，不再简单写死为固定 `T-2` 锚点，而是：

- 先核对 `净值日期`
- 优先选用日期与 `净值日期` 对齐的真实指数基准值
- 优先选用日期与 `净值日期` 对齐的真实汇率基准值
- 若确实缺少对齐基准，再按真实可得的回退路径降级

目标是让结果尽量反映最新净值，而不是把已经前移的净值继续按旧锚点重复修正。

### 13.4 监控池补充限制

`暂停申购` 的样本：

- 仍保留在主表中，便于观察
- 不进入 `限购监控池`
- 不进入 `非限购监控池`
- 不触发即时入池推送

## 14. 2026-03-25 欧美外部行情补数修订

### 14.1 修订目标

`QDII欧美` 不能只依赖集思录 `cal_tips` 和少量腾讯指数映射。

当源页没有给出足够的：

- 当前指数值
- 净值日期对应指数基准值
- 当前汇率
- 净值日期对应汇率基准值

就要引入独立外部行情 API 补齐真实输入，否则很多样本只能退回估算口径，甚至无法计算 IOPV。

### 14.2 本轮正式补数链路

本轮新增一条最小外部补数链路：

- provider 固定为 `Stooq`
- 只补指数 / 汇率输入
- 不替代集思录主列表

补数范围仅限：

- `QDII欧美`
- 且已经配置 exact symbol 映射的样本

### 14.3 补数规则

若某行缺少 `currentIndexValue`：

- 用外部 API 最新真实值补齐

若某行缺少 `baseIndexValue`：

- 用 `navDate` 对齐的真实历史收盘值补齐

若某行缺少 `currentFxRate`：

- 用外部 API 最新真实汇率补齐

若某行缺少 `baseFxValue`：

- 用 `navDate` 对齐的真实历史汇率收盘值补齐

### 14.4 真值边界

- 只有 exact 映射存在时，才允许进入外部精确补数路径
- 不允许把 ETF 代理、模糊搜索结果、猜测代码伪装成“精确指数”
- exact 映射不存在时，继续使用现有真实回退路径
- 若回退路径也不足，则继续明确显示“无法计算 / 真实输入不足”

### 14.5 本轮最低目标

至少要修复这类问题：

- 像 `恒生指数` 这种源页缺少完整基准、但外部 exact 行情可拿到的样本，应恢复 IOPV 计算
- 已经能算出的样本不能因为接入外部 API 反而回退
## 14. 2026-03-25 LOF公式分支补充

### 14.1 指数LOF / QDII亚洲

当净值日期为 `T-1日`，适用时段为收盘后到下一交易日收盘前：

`IOPV = T-1 日净值 × (1 + 指数涨跌幅) × (今日汇率 / 基准汇率)`

当净值日期为 `当日`，适用时段为交易日交易时段：

`IOPV = T 日净值`

其中 `指数涨跌幅` 直接使用集思录源侧字段，不另外替换成别的指数涨跌幅来源。

### 14.2 QDII欧美

当净值日期为 `T-2日`：

`IOPV = T-2 日净值 × (今日指数 / T-2 指数) × (今日汇率 / T-2 汇率)`

当净值日期为 `T-1日`：

`IOPV = T-1 日净值 × (今日指数 / T-1 指数) × (今日汇率 / T-1 汇率)`

这里的 `今日指数` 和 `今日汇率` 优先通过外部实时 API 抓取；若外部实时行情拿不到，再 truthfully 回退到现有源侧估算口径，不伪造实时锚点。

### 14.3 监控池限制

`暂停申购` 的样本：

- 保留在主表
- 不进入限购监控池
- 不进入非限购监控池
- 不触发即时入池推送
