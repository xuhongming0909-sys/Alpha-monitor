# Alpha Monitor UI Design System

## 1. 定位

Alpha Monitor 是金融机会监控终端，不是营销官网，也不是装饰型数据大屏。

页面设计必须优先服务三件事：

1. 发现机会
2. 判断风险
3. 继续跟踪

所有视觉、字体、排版、组件和交互选择，都必须证明自己能提升上述任一目标。不能证明的装饰应删除。

## 2. 视觉基调

采用 Bloomberg 终端式金融界面方向：高密度、数字优先、低装饰、强对齐、强状态反馈。

必须做到：

- 数据比容器更醒目。
- 表格比卡片更重要。
- 状态比氛围更重要。
- 首屏比品牌表达更重要。
- 长时间盯盘比第一眼惊艳更重要。

禁止：

- 大 hero。
- 营销式渐变背景。
- 大面积低信息留白。
- 为了高级感牺牲数据密度。
- 用图片、插画、装饰图形替代真实数据。
- 把金融监控页面做成品牌落地页或科技大屏。

## 3. 字体规范

### 3.1 字体栈

正文使用系统中文字体，保证清晰、稳定、无授权风险：

```css
font-family: "Microsoft YaHei UI", "PingFang SC", "Noto Sans CJK SC", Arial, sans-serif;
```

数字、价格、百分比、金额、代码、时间、表格数值使用等宽数字字体栈：

```css
font-family: "JetBrains Mono", "SFMono-Regular", Consolas, "Roboto Mono", monospace;
font-variant-numeric: tabular-nums;
```

不引入商业字体。除非后续文档明确批准，不通过外部字体服务加载字体。

### 3.2 字号层级

默认桌面端使用紧凑层级：

- 页面标题：18-22px。
- 区块标题：14-16px。
- 表格表头：12-13px。
- 表格正文：12-14px。
- 辅助说明：11-12px。
- 数字重点值：16-24px，必须克制使用。

禁止为了“高级感”放大无交易意义的标题。

## 4. 排版与密度

### 4.1 桌面优先

核心设计宽度以 1440px 及以上桌面屏为主。移动端保证可读、可操作，但不牺牲桌面高密度。

1440px 首屏必须同时展示：

1. 系统状态与更新时间。
2. 机会排序或核心机会列表。
3. 至少一个高密度数据区。

首屏不得被品牌标题、欢迎语、推送设置、大卡片说明占据。

### 4.2 网格

使用紧凑网格：

- 页面外边距：8-16px。
- 区块间距：8-12px。
- 面板内边距：8-12px。
- 表格单元格横向 padding：6-10px。
- 表格单元格纵向 padding：4-7px。
- 常规数据行高度：28-36px。

如果一个区域删除边框后仍能看懂，应优先减少容器感。

### 4.3 对齐

- 数字右对齐。
- 文本左对齐。
- 状态标签居中或左对齐，但必须在同列保持一致。
- 同类指标必须保持列宽、单位、精度一致。
- 排序列必须能被快速识别。

## 5. 色彩规范

### 5.1 基础色

采用深色金融终端底色：

- 页面背景：接近黑色的冷灰或炭黑。
- 面板背景：略高于页面背景，不使用强玻璃拟态。
- 分割线：低对比细线。
- 文本主色：高对比浅灰。
- 辅助文本：中低对比灰。

### 5.2 语义色

红、绿、黄只表达数据语义，不做装饰滥用：

- 红：风险、下跌、负向、异常。
- 绿：上涨、正向、机会、通过。
- 黄/橙：警告、临界、待确认。
- 蓝：链接、可操作入口、选中状态，少量使用。

同一页面不得同时使用多套涨跌颜色规则。

### 5.3 禁止项

- 大面积红色背景。
- 大面积荧光色。
- 多种渐变叠加。
- 卡片阴影作为主要层级手段。
- 颜色仅用于装饰而无状态含义。

## 6. 核心组件规范

### 6.1 状态栏

状态栏放在页面最上方，必须紧凑展示：

- 当前交易状态。
- 最近更新时间。
- 关键数据源状态。
- 自动刷新状态。
- 失败数据源数量或异常提示。

状态栏高度应控制在一行或两行内，不做大横幅。

### 6.2 机会榜

机会榜是首屏核心组件。

必须展示：

- 策略或模块来源。
- 标的名称与代码。
- 核心机会指标。
- 风险或限制字段。
- 更新时间或数据状态。
- 进入详情的操作入口。

机会榜排序必须使用现有真实字段，不得伪造综合分。若后续需要综合分，必须先补充业务规格。

### 6.3 表格

表格是默认数据展示形态。

必须做到：

- 表头 sticky 优先。
- 数字右对齐。
- 数字使用等宽数字。
- 价格、百分比、金额保留统一精度。
- 可排序列显示明确状态。
- 空值显示 `--`。
- 风险字段不得隐藏在 hover 中。

禁止：

- 把大批数据拆成宽松卡片。
- 每行加入无意义图标。
- 用多行文案挤占关键数字列。
- 为了圆角和阴影降低可扫描性。

### 6.4 筛选栏

筛选栏必须服务快速缩小范围：

- 搜索框紧凑。
- 筛选项靠近数据表。
- 常用筛选优先外露。
- 低频设置收起。

筛选栏不得占据首屏过多高度。

### 6.5 Tab

Tab 用于策略模块切换：

- 标签文字短。
- 当前项清晰。
- 未读、异常、数量可作为紧凑徽标展示。
- 不使用大按钮式导航。

### 6.6 错误、空状态、加载状态

状态展示必须紧凑，不破坏页面密度：

- 加载：骨架行或状态文本，不使用大动画。
- 空状态：说明当前没有机会或数据，不写营销文案。
- 错误：展示失败数据源、失败原因、刷新入口。

错误状态不能伪装成空数据。

## 7. 页面结构原则

推荐结构：

1. 顶部状态栏。
2. 首屏机会指挥区。
3. 策略模块 tab。
4. 当前策略高密度表格。
5. 次级设置与推送配置。

推送设置、说明文档、公式解释属于次级内容，默认不抢首屏。

## 8. 内容与文案

文案使用工具型语言，不使用营销型语言。

推荐：

- `数据已更新 09:31:12`
- `发现 12 个候选机会`
- `3 个数据源异常`
- `按溢价率升序`

禁止：

- `开启你的 Alpha 之旅`
- `智能洞察市场未来`
- `高级金融驾驶舱`
- 无实际状态含义的口号

## 9. 验收清单

任何 UI 设计或实现进入编码前，必须检查：

- 是否符合“发现机会 / 判断风险 / 继续跟踪”。
- 1440px 首屏是否展示状态、机会排序和高密度数据区。
- 是否避免大 hero、宽松卡片、营销式渐变。
- 表格数字是否右对齐并使用等宽数字。
- 价格、百分比、金额是否精度一致。
- 红、绿、黄是否只表达数据语义。
- 错误状态是否不同于空状态。
- 推送设置是否没有抢占首屏。
- 是否使用真实接口和真实数据，不使用假数据占位。
- 是否能被长期盯盘使用，而不只是截图好看。

不满足清单的 UI 方案，不进入编码阶段。
