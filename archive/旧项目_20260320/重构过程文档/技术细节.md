# Alpha Monitor Tech Details

这份文档保留给实现和重构过程使用，不给 PM 日常阅读。

## 1. 当前技术背景

- 当前系统是 `Node.js + Python` 双运行时。
- 当前根入口仍是：
  - `启动服务.js`
  - `数据调度.py`
  - `页面入口.html`
- 当前重构方向是：
  - `共享基础/`
  - `数据抓取/`
  - `策略中心/`
  - `页面展示/`
  - `消息推送/`

## 2. 保留的技术约束

- 必须先改文档，再改代码。
- `config.yaml` 是长期配置唯一真相源。
- UI 维护的数据继续走运行态 JSON，不混入长期配置。
- `数据抓取/*` 与 `策略中心/*` 采用插件式目录，插件之间禁止互相引用。
- 抓取层到策略层必须通过 `The Bus`。
- 修改 `消息推送/` 推送样式时，不应要求修改其他职责目录。

## 3. 迁移期兼容要求

- `启动服务.js` 迁移期保留 Node 服务入口职责。
- `数据调度.py` 迁移期保留 CLI facade 职责。
- `页面入口.html` 迁移期保留页面壳层职责。
- `命令工具/*` 可以暂时保留，但后续改为内部适配器或迁入插件目录。

## 4. 现阶段仍需关注的技术事项

- AH / AB 当前“溢价”口径与 PM 直觉公式可能不一致，已经在 `规格说明.md` 中按“当前代码事实”登记。
- 打新“中签查看”是确认过的目标需求，但当前代码还没有单独事件筛选。
- 可转债理论定价使用的是“纯债价值 + 美式期权”的组合口径，依赖历史波动率和 10 年期国债收益率。
- 自定义套利监控目前已经兼容“换股 + 现金”和“纯现金”。

## 5. 后续开发时要同步看的文件

- PM 需求变动：先看 `需求手册.md`
- 公式和阈值变动：先看 `规格说明.md`
- 架构和迁移实现：再看 `施工计划.md`、`任务清单.md`

## 6. 当前推荐的实现核对顺序

1. 先核对 `需求手册.md` 有没有业务变化
2. 再核对 `规格说明.md` 的公式编号有没有变化
3. 再改代码
4. 改完后手动比对“代码 vs SPEC 条目编号”
