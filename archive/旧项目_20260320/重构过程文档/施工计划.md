## Summary

本次重构的核心目标不是新增单一功能，而是在不损失现有运行能力的前提下，把 Alpha Monitor 重构成符合宪法约束的职责分层系统：先建立 `config.yaml` 统一配置契约，再建立 `数据抓取/`、`策略中心/`、`页面展示/`、`消息推送/` 四个核心目录，随后以“兼容入口 + 渐进迁移”的方式把现有 `启动服务.js`、`页面入口.html`、`数据调度.py` 和 `命令工具/` 中的耦合逻辑逐步迁出。

技术路线采用“外层兼容、内层替换”的方式推进：在根目录保留现有启动入口和主要命令作为兼容壳层，在新目录中逐步建立配置层、抓取层、策略层、表现层和推送层。每完成一个迁移波次，都必须让现有命令继续可运行，并用现有检查命令验证没有破坏核心能力。

## Technical Context

**Language/Version**: Node.js >= 18, Python 3.x（当前仓库已存在 CPython 3.13 运行痕迹）  
**Primary Dependencies**: Express、cors、AkShare、pandas、requests、beautifulsoup4、numpy  
**Storage**: SQLite 数据库、运行态 JSON、`.env` 环境变量、待新增 `config.yaml`  
**Testing**: `npm run check`、`node 冒烟检查.js`、`python 数据调度.py exchange-rate`、`python 数据调度.py ah`、必要时补充模块级回归检查  
**Target Platform**: Windows 本地运行环境优先，浏览器看板 + Node 服务 + Python 抓取脚本  
**Project Type**: 单仓库双运行时 Web dashboard + data worker system  
**Performance Goals**: 股价与汇率可实时刷新；并购公告每日 00:00 刷新；默认 08:00 完整推送；历史数据库每日增量更新  
**Constraints**: 不能破坏现有主要 `/接口/*` 能力；历史数据库更新必须增量维护；重构期间根入口必须保持可运行；核心逻辑必须补齐中文注释  
**Scale/Scope**: 一个仓库内统一承载 AH/AB、可转债、并购、打新、分红、自定义监控与企业微信推送

## Constitution Check

*GATE: Must pass before implementation starts. Re-check after each major migration wave.*

- `Gate 1 - 配置先行`: 第一个实施步骤必须创建 `config.yaml`，并建立统一配置加载层；任何新增参数不得继续散落在 `.env`、JSON 或硬编码常量中。
- `Gate 2 - 核心目录先行`: 在开始迁移业务逻辑前，必须先建立 `数据抓取/`、`策略中心/`、`页面展示/`、`消息推送/` 四个核心目录，以及承接公共能力的 `共享基础/` 或 `core/`。
- `Gate 3 - 无损迁移`: 根目录 `启动服务.js`、`数据调度.py`、`页面入口.html` 在迁移中保留兼容入口职责，不允许在目录未稳定前直接删除现有入口。
- `Gate 4 - 单向依赖`: 数据抓取层只负责抓取和标准化；策略层只消费抓取结果与配置；表现层只消费策略结果；推送层只消费策略/表现层产物，禁止反向耦合。
- `Gate 5 - 中文注释`: 新增模块与迁移后的核心逻辑必须补齐中文注释，解释职责、边界与业务意图。
- `Gate 6 - 渐进验证`: 每个阶段结束后必须至少通过 `npm run check`、`python 数据调度.py exchange-rate`、`python 数据调度.py ah`；若阶段涉及推送、并购或历史库，还必须补充对应验证。

**Gate Result**: 通过。当前计划不要求越过宪法例外条款，且完整遵守“先配置、先建目录、渐进迁移、保持可运行”原则。

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| 保留根目录 `启动服务.js` / `数据调度.py` / `页面入口.html` 作为兼容壳层 | 渐进式重构必须保证现有命令和页面持续可运行 | 直接删除旧入口会让迁移变成一次性替换，风险过高 |
| 暂时保留 `命令工具/` 与新目录并存 | 需要逐市场迁移，不能一次性替换全部抓取脚本 | 立刻整体搬迁会放大回归面，且不利于定位问题 |

## Migration Strategy

### 总体策略

本次施工采用六条硬规则：

1. 先建新结构，再搬旧逻辑，不做“边改边散落”的混合式改法。
2. 先抽配置，再抽业务；没有进入 `config.yaml` 的长期参数，视为迁移未完成。
3. 先做兼容壳层，再做内部替换；外部命令和主要 API 在迁移期间保持稳定。
4. 一次只迁一个职责维度或一个业务簇，避免同时重写抓取、策略、推送三层。
5. 历史数据库只做增量维护，不做破坏性整库替换。
6. 每一波迁移结束都要运行最小验证命令，失败则停止进入下一波。

### 兼容原则

- `启动服务.js` 在迁移期间仅允许逐步缩减职责，不允许新增大段业务实现。
- `数据调度.py` 在迁移期间仅允许作为 CLI 分发壳层，逐步转向调用 `数据抓取/` 与 `策略中心/` 中的新模块。
- `页面入口.html` 在迁移期间保持页面可访问，逐步把 JS 逻辑转移到 `页面展示/` 中的独立脚本或模板资源。
- 现有 `运行数据/共享基础/*.json` 先按“配置/状态”重新分类，迁移时只改变归属，不先改变业务语义。

## Detailed Phases

### Phase 1: 配置启动与目录骨架

**Goal**: 先建立统一配置契约和四大核心目录，冻结未来所有新代码的落点。

**Step 1.1 - 创建 `config.yaml` 并完成第一次参数归档**

- 创建根目录 `config.yaml`，按 `app / data_fetch / strategy / presentation / notification / storage / logging` 分段。
- 首轮提取以下长期参数：
  - 服务端口、主机、共享数据目录、Python 路径声明
  - 实时刷新节奏、每日数据库更新时间、并购公告 00:00 刷新规则
  - 默认 08:00 完整推送基线
  - AH/AB 前三推送口径
  - 可转债六类推送分类定义
  - 并购当日分析限制
  - 分红登记日提醒规则
  - 打新覆盖范围与事件范围
- 保留敏感值使用环境变量占位，但把字段名和用途登记进 `config.yaml`。

**Step 1.2 - 建立新目录骨架**

- 创建 `共享基础/`、`数据抓取/`、`策略中心/`、`页面展示/`、`消息推送/`。
- 为每个目录放置最小入口文件或 README/模块说明，声明职责边界。
- 建立 `共享基础/config` 与 `共享基础/paths` 作为最先落地的公共层。

**Step 1.3 - 建立配置加载器与兼容回退**

- Node 侧新增统一配置读取模块，优先从 `config.yaml` 读取，必要时兼容旧 `.env`。
- Python 侧新增统一配置读取模块，禁止后续业务模块自行调用 `os.getenv`。
- 当前 `.env` 暂不删除，只作为敏感值注入源。

**No-loss Migration**

- 原有 `启动服务.js` 和 `数据调度.py` 先改为调用新配置加载器，但保留旧行为回退。
- 此阶段不移动业务逻辑，只移动配置来源。

**Validation**

- `npm run check`
- `python 数据调度.py exchange-rate`
- `python 数据调度.py ah`
- 手动确认 `config.yaml` 与旧配置能共同驱动服务启动

### Phase 2: 共享基础能力抽离

**Goal**: 先把不会改变业务语义的公共能力从大文件中拆出，为后续迁移铺路。

**Scope**

- 路径策略：从 `数据库路径.py` 和 `启动服务.js` 中抽出共享路径解析
- 时间策略：交易时段、上海时区、日期格式化
- 运行态读写：JSON 读取、写入、状态存储适配
- 通用模型：成功/失败响应、统一数据包装

**No-loss Migration**

- `数据库路径.py` 保留原接口，但内部委托给 `共享基础/paths`
- `启动服务.js` 中基础函数改为从 `共享基础/` 引用，不改变 API 输出格式
- `数据调度.py` 中响应包装函数逐步委托给共享模块

**Validation**

- `npm run check`
- `python 数据调度.py exchange-rate`
- `python 数据调度.py ah`
- 抽查 `/接口/health`

### Phase 3: Notification 域先行迁移

**Goal**: 先从 `启动服务.js` 中拆出最清晰的一条职责链：推送配置、推送拼装、推送执行、定时调度。

**Scope**

- 创建 `消息推送/企业微信/`
- 创建 `消息推送/调度/`
- 创建 `消息推送/摘要/`
- 创建 `消息推送/并购报告/`

**迁移内容**

- `推送配置.json` 与 `推送运行状态.json` 的读取和保存逻辑迁出 `启动服务.js`
- 企业微信 Markdown 发送逻辑迁出 `启动服务.js`
- 主推送摘要拼装逻辑迁出 `启动服务.js`
- 并购报告推送逻辑迁出 `启动服务.js`
- 定时触发逻辑迁出 `启动服务.js`

**No-loss Migration**

- `启动服务.js` 暂时仍保留 `/接口/push/*` 路由，但路由内部只调用 `消息推送/` 服务。
- 推送时间仍允许 UI 修改，但必须明确这是运行态数据，不回写长期业务配置。

**Validation**

- `npm run check`
- 手动调用 `/接口/push/config`
- 手动调用 `/接口/push/wecom` 的 dry-run 或受控推送验证
- 并购当日无公告时验证空推送路径

### Phase 4: Data Fetch 域迁移

**Goal**: 把“抓数据”和“算策略”彻底分开，先处理抓取层。

**Scope**

- `数据抓取/market/`: 汇率、搜索、单股价格、AH/AB 原始行情
- `数据抓取/打新s/`: IPO、可转债申购源
- `数据抓取/分红/`: 分红与价格源
- `数据抓取/并购公告/`: 公告抓取源
- `数据抓取/可转债/`: 可转债原始行情、正股历史、财务指标
- `数据抓取/history/`: 历史数据库增量同步入口

**迁移原则**

- 原有 `命令工具/*.py` 先按职责拆成“原始抓取适配”与“库写入适配”两层。
- `数据调度.py` 不再直接混写抓取与策略，只做 action 到新抓取/策略服务的分发。
- `市场配对.py` 拆为“配对源抓取”和“配对持久化同步”。

**No-loss Migration**

- 旧 `命令工具/*.py` 可保留文件名，但内部改为调用 `数据抓取/` 新模块。
- 所有现有 CLI 命令和 npm script 保持名称不变。

**Validation**

- `python 数据调度.py exchange-rate`
- `python 数据调度.py ah`
- `python 数据调度.py ab`
- `python 数据调度.py ipo`
- `python 数据调度.py bonds`
- `python 数据调度.py merger`

### Phase 5: Strategy 域迁移

**Goal**: 把所有业务判断、排名、筛选、收益率逻辑从抓取层和入口层中拆出来。

**Scope**

- `策略中心/premium/`: AH/AB 溢价排序、前三推送规则
- `策略中心/可转债/`: 双低、权债收益率、回售套利、涨停套利、转股套利、退市博弈
- `策略中心/monitor/`: 自定义套利监控收益计算与纯现金兼容
- `策略中心/分红/`: 登记日提醒选择
- `策略中心/打新s/`: 当日申购、中签查看、缴款、上市事件选择
- `策略中心/并购公告/`: 当日公告筛选与可分析集合确定

**No-loss Migration**

- 先抽纯函数，保持输入输出结构与旧系统一致。
- `启动服务.js` 和 `数据调度.py` 在过渡期只做“调用策略模块并返回旧格式结果”。

**Validation**

- 对比迁移前后关键结果集样本数量与核心字段
- 验证 AH/AB 推送前三结果仍符合旧逻辑
- 验证可转债六类摘要仍能生成
- 验证自定义监控纯现金模式结果不变

### Phase 6: Presentation 域迁移

**Goal**: 把 API 输出整形、前端视图模型、页面脚本从根入口中拆出。

**Scope**

- `页面展示/接口/`: `/接口/*` 路由控制器与响应装配
- `页面展示/视图模型/`: 看板、表格、推送摘要使用的数据模型
- `页面展示/看板/`: 前端独立脚本与状态管理
- `页面展示/模板/`: 页面模板与静态资源

**迁移内容**

- `启动服务.js` 的 route handler 迁至 `页面展示/接口/`
- `页面入口.html` 中大段脚本迁至独立 JS 资源
- 页面仅保留最小 HTML 模板与资源挂载

**No-loss Migration**

- `页面入口.html` 仍保留入口文件名，避免静态资源路径变化影响使用
- `/接口/*` 路径保持不变，先替换内部实现，不改变外部契约

**Validation**

- `npm run check`
- 页面可正常打开
- 核心 tab：AH、AB、并购、套利监控、分红 逐项人工回归
- 即时推送配置 UI 仍可使用

### Phase 7: 历史库与运行态数据边界整理

**Goal**: 把“长期配置”和“运行态状态”最终分清，消除当前 JSON 既像配置又像状态的问题。

**迁移原则**

- `config.yaml` 保留长期业务参数
- 运行态 JSON 仅保留：
  - 推送时间等界面可维护数据
  - 用户维护的自定义监控列表
  - 分红组合名单
  - 推送记录、缓存、当日报告缓存

**具体动作**

- 列出全部现存 JSON 文件，并逐个标记“配置/状态/缓存/seed”
- 把误归为 JSON 配置的长期参数迁回 `config.yaml`
- 为仍然保留的运行态 JSON 建立统一读写层和字段说明

**Validation**

- 校验 UI 修改数据仍可保存
- 校验长期参数不再从运行态 JSON 偷读
- 校验历史数据库路径与 profile 行为不变

### Phase 8: 入口收口与遗留清理

**Goal**: 让根目录入口恢复成真正的薄壳层，并清理过渡适配代码。

**完成标准**

- `启动服务.js` 只负责启动、装配、挂载路由和调度
- `数据调度.py` 只负责命令分发
- `页面入口.html` 只负责模板和资源装配
- `命令工具/` 中只保留内部工具或兼容脚本
- 核心逻辑中文注释补齐

**No-loss Migration**

- 只在新目录稳定且验证通过后删除旧实现片段
- 每删除一段旧逻辑，都要确认已有新模块完全接管

**Validation**

- `npm run check`
- `python 数据调度.py exchange-rate`
- `python 数据调度.py ah`
- `python 数据调度.py ab`
- 关键推送回归
- 页面人工巡检
