# Alpha Monitor Current State Audit

**Feature Branch**: `001-monitor-refactor`  
**Created**: 2026-03-20  
**Scope**: 基于《Alpha Monitor Constitution》对当前仓库进行现状审计，识别领域边界、配置分散点、耦合热点与重构风险。  
**Audit Baseline**: [项目宪法.md](C:/Users/93724/Desktop/Alpha%20monitor/项目宪法.md)

## 1. 审计结论摘要

当前仓库已经具备多市场监控原型能力，但整体仍是“根目录大一统 + 脚本型拼装”的状态，尚未满足宪法中的“领域解耦优先”和“配置即契约”。

最核心的问题有四个：

1. `启动服务.js` 已经演化为单文件中枢，混合了服务启动、配置读取、定时调度、市场数据聚合、套利监控计算、分红组合管理、企业微信推送、并购 AI 报告生成与全部 HTTP 路由。
2. `页面入口.html` 是单文件前端应用，混合了页面结构、样式、状态管理、搜索、排序、表格渲染、表单处理、推送配置 UI 与各业务模块交互。
3. `数据调度.py` 是 Python 侧总入口，混合了搜索、汇率获取、AH/AB 实时快照、历史回写、并购抓取调度、分红查询、CLI 路由分发。
4. 当前不存在 `config.yaml`。配置散落在 `.env`、`process.env`、Python `os.getenv`、运行态 JSON 文件、代码内常量、前端默认值中，导致修改路径不清晰。

结论：当前项目可运行，但不是按“数据抓取 / 策略逻辑 / 页面显示 / 消息推送”划分的系统，而是按“入口文件 + 辅助脚本 + 单页 HTML”堆叠出来的系统。

## 2. 当前能力版图

从已实现能力看，项目已经覆盖以下业务域：

- AH / AB 溢价快照与历史溢价分布
- A 股新股申购监控
- 可转债申购监控
- 可转债套利机会筛选
- 并购 / 私有化 / 吸收合并机会抓取
- 自定义套利监控
- 分红组合与股权登记日提醒
- 企业微信定时与即时推送
- 并购公司 AI 摘要报告

从运行时形态看，当前是双运行时系统：

- Node.js 负责 Web 服务、调度、推送、前端静态页和部分业务组装
- Python 负责市场抓取、历史数据库、计算与部分 CLI 能力

## 4. 违反“领域解耦”原则的具体证据

下面按宪法中的四个目标领域来审计：`data_fetch / strategy / presentation / notification`。

### 4.1 `启动服务.js` 同时承担了至少 7 类职责

`启动服务.js` 是当前最严重的违例文件。

它同时做了：

- 配置层：读取 `.env`、解析 `process.env`、决定数据目录与 Python 可执行程序。
- 数据抓取编排层：`DATASETS`、`callDataCore()`、`getDataset()`、`runIntradayRefreshCycle()`、`runDailySync()`。
- 策略层：`recalculateMonitor()`、`cbArbOpportunitySets()`、`getOverviewFromDatasets()`。
- 状态持久化层：读写 `自定义监控.json`、`分红组合.json`、`推送配置.json`、`推送运行状态.json`、`并购公司报告.json`。
- 通知层：`buildWeComSummaryMarkdown()`、`sendWeComMarkdown()`、`pushSummaryToWeCom()`、`runMainPushIfNeeded()`。
- AI 报告层：`callDeepSeekChatCompletion()`、`buildMergerCompanyReportMarkdown()`、`ensureMergerCompanyReport()`。
- 表现层 API：`/接口/market/*`、`/接口/dividend`、`/接口/monitors`、`/接口/push/*`、`/接口/merger/*`。

这意味着 `启动服务.js` 既做“抓取”，又做“策略”，还做“推送”和“页面输出”。它直接违反了宪法要求的单一职责与单向依赖。

最典型的交叉案例：

- 同一文件中，`collectSummaryDatasets()` 负责抓取各类市场数据，`cbArbOpportunitySets()` 负责套利规则筛选，`buildWeComSummaryMarkdown()` 负责消息格式化，`sendWeComMarkdown()` 负责真正推送。
- 同一文件中，`getAllMonitors()` 负责获取监控标的并回填实时股价，`recalculateMonitor()` 负责收益率计算，随后 `/接口/monitors` 与推送逻辑又直接消费这些结果。

### 4.2 `页面入口.html` 同时承担页面显示、前端状态机和业务控制器职责

`页面入口.html` 违反的是“页面显示层不得自行承担过多业务流程”。

它当前同时包含：

- 页面结构与样式定义
- 全局状态 `state`
- 市场数据搜索与本地候选池构建
- 各模块排序、筛选、分页
- 并购报告加载控制
- 分红组合管理交互
- 推送配置保存与即时推送交互
- 自动刷新调度 `startAutoRefresh()`

这意味着前端不是薄表现层，而是一个单文件业务控制台。尤其 `searchStocks()` 会从 `AH / AB / 可转债 / 分红 / 套利监控` 多个状态池拼接本地搜索候选，说明前端已经知道多个领域的数据结构和组合方式。按宪法要求，这类跨域组合应该下沉到清晰的服务接口或领域适配层，而不是留在页面文件中。

### 4.3 `数据调度.py` 同时承担抓取、计算、持久化和 CLI 总路由职责

`数据调度.py` 是 Python 侧最典型的“多职责聚合文件”。

它当前同时做了：

- 基础抓取：腾讯行情、汇率、搜索
- 领域计算：AH/AB 溢价计算、分位数与历史统计拼接
- 持久化：`premium_history.db` 历史回写
- 模块编排：直接导入 `get_ipo / get_bonds / get_dividend / get_merger_arbitrage / 抓取可转债 / sync_cb_stock_history`
- CLI 分发：`main()` 根据 action 路由所有数据能力

这说明 Python 侧并不是 `data_fetch` 下的整洁分层，而是“一个大入口直接调用若干脚本”。这对后续解耦非常不利，因为任何新市场类型或策略都倾向继续向 `数据调度.py` 追加 action。

### 4.4 `命令工具/抓取可转债.py` 同时承担抓取、历史补数、财务指标计算和套利策略职责

这是 Python 侧最复杂的单文件业务模块之一。

它当前同时做了：

- 从多个源抓取可转债实时与静态数据
- 补齐正股历史价格
- 计算波动率
- 计算 ROE / 负债率等财务指标
- 计算纯债价值、看涨期权、看跌期权、理论价格、理论溢价率
- 依据排除规则过滤数据
- 输出面向展示/推送的完整结果

按宪法要求，这至少应拆成：

- `data_fetch`: 可转债原始行情、正股历史、财务指标抓取
- `strategy`: 双低、转股套利、回售套利、理论溢价率计算
- `presentation`: 最终统一输出模型

当前全部塞在同一文件，修改任一策略阈值都需要深入大型抓取文件，修改路径非常不简洁。

### 4.5 `命令工具/市场配对.py` 同时承担抓取、匹配规则、手工映射和数据库同步职责

`市场配对.py` 当前同时包含：

- 腾讯报价抓取
- Smartbox 搜索
- AH / AB 配对解析
- 名称一致性规则
- 大量手工映射常量
- 旧数据 seed 兼容
- SQLite 初始化与同步日志

这说明“配对发现”并没有被拆成“数据抓取”和“规则匹配”两个层次，而是数据源、规则、数据修复、落库全都在一起。

### 4.6 `命令工具/抓取新股.py` 与 `命令工具/抓取新债.py` 同时承担抓取、状态规则和持久化职责

这两个文件都不是纯抓取器，而是同时做了：

- 访问 AkShare
- 字段清洗
- 日期规则纠偏
- 状态判定（即将申购 / 申购中 / 待缴款 / 待上市 / 已上市）
- 写入 `subscription_history.db`
- 输出面向前端的结构化结果

这意味着“申购事件数据抓取”和“申购状态策略”还没有分离。

### 4.7 `命令工具/抓取分红.py` 同时承担分红抓取、股价抓取和收益率计算职责

`抓取分红.py` 当前同时做了：

- CNINFO 分红抓取
- 腾讯实时报价抓取
- 币种与币种符号判定
- 股息率计算
- CLI 动作分发

这本质上把“数据抓取”和“策略计算”合并在一个文件里。未来如果需要区分“现金分红提醒”“高股息筛选”“登记日前提醒规则”，会继续加重耦合。

### 4.8 `命令工具/抓取并购套利.py` 同时承担抓取、文本规则与价格补充职责

`抓取并购套利.py` 当前同时做了：

- 从巨潮抓取公告
- 多关键词并发搜索
- 核心公告识别
- 子公司内部重组过滤
- 非上市主体交易过滤
- 已结束交易过滤
- 从标题中解析要约价格
- 批量补实时股价

这意味着并购领域也没有拆成“公告抓取器”“规则过滤器”“并购机会策略”。

## 5. 哪些代码明显违反了“抓取与推送分离”

用户特别关心“哪些文件同时干了抓取和推送的活”，当前最明显的是：

### 5.1 `启动服务.js`

`启动服务.js` 同时包含：

- 数据抓取编排：`getDataset()`、`collectSummaryDatasets()`
- 推送格式化：`buildWeComSummaryMarkdown()`
- 推送执行：`sendWeComMarkdown()`、`pushSummaryToWeCom()`、`pushMergerReportToWeCom()`

也就是说，这个文件既知道“怎么拿到数据”，又知道“怎么生成消息”，还知道“怎么把消息发出去”。

### 5.2 `启动服务.js` + 运行态 JSON 的组合

推送配置与推送记录由 `启动服务.js` 直接读写：

- `运行数据/共享基础/推送配置.json`
- `运行数据/共享基础/推送运行状态.json`

这使得推送模块并没有独立边界，而是被服务入口直接控制和持久化。

## 6. 违反“配置即契约”原则的具体证据

### 6.1 当前没有 `config.yaml`

仓库里不存在 `config.yaml`，因此当前完全不满足“单一配置入口”。

### 6.2 配置来源至少分散在 5 个地方

当前配置来源包括：

1. `.env` / `.env.example`
2. Node `process.env`
3. Python `os.getenv`
4. 运行态 JSON
5. 代码中的硬编码常量

### 6.3 环境变量散落

已发现的环境变量包括但不限于：

- `PORT`
- `HOST`
- `WECOM_WEBHOOK_URL`
- `PUSH_HTML_URL`
- `ALPHA_MONITOR_HTML_URL`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `ALPHA_SHARED_DATA_DIR`
- `ALPHA_DB_PROFILE`
- `PYTHON_BIN`
- `JISILU_COOKIE`

这些变量目前没有统一的配置契约说明，也没有集中归档。

### 6.4 运行态 JSON 实际承担了配置角色

以下 JSON 文件不是纯状态，而是已经承担了业务配置责任：

- `运行数据/共享基础/推送配置.json`: 推送开关、时点、模块选择、并购推送时点
- `运行数据/共享基础/自定义监控.json`: 自定义监控规则、价格、币种、安全边际
- `运行数据/共享基础/分红组合.json`: 分红监控股票池

其中至少前两个都属于“配置即契约”中的正式业务配置，未来应进入 `config.yaml` 或统一配置域，而不是继续以散落 JSON 形式存在。

### 6.5 大量硬编码常量

当前硬编码常量广泛存在，例如：

- 各数据集刷新频率
- 推送默认时间
- 交易时段窗口
- 并购摘要最大字数
- 各类套利规则阈值
- 前端默认分页、排序、自动刷新节奏

这直接导致“修改路径不止一条”，与宪法中的极简修改目标冲突。

## 7. 违反“极简修改路径”原则的具体证据

当前如果要改一个简单需求，往往要动多个位置。

### 7.1 修改推送时间

当前至少涉及：

- `启动服务.js` 中默认推送时间逻辑
- `运行数据/共享基础/推送配置.json` 中现有运行配置
- `页面入口.html` 中推送 UI 默认值

### 7.2 新增一个推送板块

当前至少涉及：

- `启动服务.js` 中 `normalizePushConfig()`
- `启动服务.js` 中 `buildWeComSummaryMarkdown()`
- `页面入口.html` 中推送复选框 UI
- `页面入口.html` 中读取/保存推送模块逻辑

### 7.3 调整可转债套利筛选口径

当前至少涉及：

- `命令工具/抓取可转债.py` 中计算与过滤逻辑
- `启动服务.js` 中 `cbArbOpportunitySets()` 的推送摘要规则
- `页面入口.html` 中可转债前端排序/展示逻辑

### 7.4 调整并购监控规则

当前至少涉及：

- `命令工具/抓取并购套利.py` 中关键词和过滤规则
- `启动服务.js` 中并购报告聚合与推送
- `页面入口.html` 中并购表格与报告加载交互

这些都说明当前“一个需求只改一条路径”的目标还没有实现。

## 8. 中文注释现状

当前代码库并未达到“每一段核心逻辑都要有中文注释”的要求。

现状特点如下：

- 部分 Python 脚本有模块头注释和局部中文注释。
- 大量关键函数只有英文函数名，没有中文说明其业务语义。
- `启动服务.js` 中大量复杂逻辑几乎没有中文注释，尤其是调度、推送、监控收益计算、并购报告缓存部分。
- `页面入口.html` 的前端脚本体量很大，但几乎缺少系统性的中文注释。

结论：中文注释要求目前整体不达标，后续重构必须把“补注释”视为结构重构的一部分，而不是最后的文案修饰。

## 9. 当前相对健康的部分

并非所有代码都完全混乱，下面这些部分相对接近后续可复用基座：

- `数据库路径.py`: 运行态数据目录策略较集中，适合作为未来 `共享基础/path_policy` 的起点。
- `命令工具/溢价历史库.py`: 职责相对单一，主要是 AH/AB 历史数据库。
- `命令工具/打新历史库.py`: 职责相对单一，主要是新股/新债申购历史数据库。
- `命令工具/股票价格历史库.py`: 职责相对单一，主要是正股价格历史数据库。
- `冒烟检查.js`: 基础健康检查简单明确。

这些文件更像“基础设施层”，后续可以保留并重命名迁移。

## 10. 现状对重构目标的影响

针对用户提出的目标“整合 A 股、B 股、港股、可转债、自定义监控、套利机会监控、打新、分红，并统一通过企业微信推送”，当前代码的现实含义是：

- 大部分能力已经存在原型，但分散在不同脚本和两个运行时中。
- 统一推送已经有雏形，但推送内容与策略逻辑强耦合在 `启动服务.js`。
- 自定义监控、分红、并购报告都走了“业务状态 JSON”路线，未来很容易继续扩散配置。
- 前后端都已有大量“领域知识硬编码”，不先拆边界，后续功能越加越难改。

因此，后续规格说明必须先定义清楚：

- 哪些属于抓取域
- 哪些属于策略域
- 哪些属于展示域
- 哪些属于推送域
- 哪些参数进入 `config.yaml`
- 哪些 JSON 只是状态，不再承担配置职责

## 11. 重构前的最低事实基线

后续 `SPEC.md` 应默认承认以下现状事实：

- 当前系统是 Node + Python 双运行时。
- 当前前端是单文件页面。
- 当前后端入口是单文件 Express 服务。
- 当前 Python 数据入口是单文件 action 分发。
- 当前支持的主要市场能力已经可用，但未按领域解耦。
- 当前推送目标唯一明确的是企业微信。
- 当前不存在统一配置中心。

## 12. 对后续规格编写的约束建议

为了符合宪法，后续 `SPEC.md` 不能只是“把文件拆文件夹”，而必须明确回答以下结构问题：

1. 抓取层是否按市场拆分，还是按能力拆分。
2. 策略层是否允许跨市场共用一套信号模型。
3. 推送层是“统一摘要推送”，还是“按主题独立推送”。
4. 自定义监控规则哪些属于用户配置，哪些属于系统运行态。
5. 当前 `运行数据/共享基础/*.json` 中哪些迁入 `config.yaml`，哪些保留为状态文件。
