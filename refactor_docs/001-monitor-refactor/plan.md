## 1. 本次目标

本次聚焦解决 4 个前端问题：

1. `推送设置` 当前交互不可信，需改成页面末尾的简洁单行编辑区，并确保保存后立即回显。
2. `转债套利` 顶部固顶摘要区占用过大，需压缩成更紧凑的小栏目。
3. 页面整体字体、间距和桌面利用率偏小，导致视觉体验差、长表难看。
4. `转债套利` 的分栏方式参考 `AH / AB` 的秩序感，但整体密度要更高、更省空间。

本次完成标准：

1. 首页顶部不再放独立推送卡片，推送设置统一放到整页最末尾。
2. 推送设置读取、保存、保存后回显都正常，用户能明确看到当前生效时间。
3. 转债套利首屏不再被三张大摘要卡占满，主表更早进入视野。
4. 桌面端整体字号、tab、表头、表格内容、输入框与按钮可读性明显提升。
5. 页面保持黑红科技风，不新增接口、不改业务公式、不改计算口径。

## 2. 本次不改的内容

这次不主动修改以下内容：

- `/api/*` 接口路径与返回口径
- 推送调度器执行策略
- Python 数据抓取与策略计算模块

如果实施中发现必须改动这些内容，会先回到文档更新流程。

## 3. 现状判断

根据当前页面实现与用户反馈，问题主要集中在展示层：

1. `dashboard_template.html` 仍把推送设置放在首页顶部右侧，占据首屏空间。
2. `dashboard_page.js` 中 `renderPushSettings()` 只在输入框为空时回填值，保存后的状态反馈也偏弱，容易让用户误判“没有生效”。
3. `renderConvertibleBondPanel()` 当前直接渲染 3 张大摘要卡，每张 5 条，纵向空间占用过重。
4. 全局字体、表头和表格内容字号偏小，桌面端可用宽度也不够激进，导致长表阅读体验差。

## 4. 实施假设

本计划先采用下面这组默认假设：

1. `推送设置` 是整个仪表盘的全局配置，放在整页所有 tab 内容之后的页面末尾。
2. 本轮不新增“启用/禁用推送”的单独开关，只保证现有 3 个时间设置的读取、保存与回显可靠。
3. `转债套利` 摘要继续保留 3 组信息，但每组仅保留更少条目，并改成紧凑分栏。
4. AH / AB 的页面结构继续保持不变，本轮只让 `转债套利` 在布局语言上向它靠拢。

目标：先收口首页大结构，释放首屏空间。

### Phase C: 推送设置交互修复

目标：让推送设置真正“可读、可改、可确认已生效”。

计划修改：

- `presentation/dashboard/dashboard_page.js`

要做的事：

1. 推送设置加载时总是按接口返回值刷新输入框
2. 保存成功后使用服务端返回值重新渲染页面状态
3. 明确展示当前主推送与收购私有专报状态
4. 保留现有接口，不新增前端专用 API

完成标准：

- 用户保存后能立刻看到最新时间与状态文案，不再怀疑“没有生效”。

### Phase D: 转债套利布局收口

目标：压缩转债套利顶部摘要区，让主表更早进入阅读区域。

计划修改：

- `presentation/dashboard/dashboard_page.js`
- `presentation/templates/dashboard_template.html`

要做的事：

1. 把 3 张大摘要卡改成更紧凑的栏目
2. 每组摘要减少条目数，降低纵向高度
3. 摘要样式参考 AH / AB 的结构秩序，但保持更高密度
4. 加强主表与摘要之间的层次感

完成标准：

- 转债套利页面打开后，无需长滚动就能快速进入主表区域。

### Phase E: 可读性与桌面利用率优化

目标：统一提升全页阅读体验。

计划修改：

- `presentation/templates/dashboard_template.html`

要做的事：

1. 提升全局字号、标题字号、tab 尺寸、表头与表格文字大小
2. 调整表格行高、卡片内边距、按钮与输入框高度
3. 提升桌面端页面宽度利用率，同时保持移动端可滚动
4. 保持现有黑红风格，不做风格重置

完成标准：

- 电脑端明显更易读，可转债页长表不再显得拥挤压抑。

## 6. 风险与处理

### 风险 1：推送设置视觉变简洁后，用户误以为丢失了功能

处理方式：

- 保留状态文案
- 保留 3 个时间框、保存按钮、刷新按钮
- 保存后即时 toast 提示 + 表单回写

### 风险 2：字号整体放大后，移动端被压坏

处理方式：

- 同步调整断点与纵向堆叠
- 保留长表横向滚动

### 风险 3：转债摘要过度压缩后信息不够用

处理方式：

- 保留 3 组摘要类别
- 保留最关键字段，但减少单组条目数和装饰面积
