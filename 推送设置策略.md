# Alpha Monitor 推送设置策略

## 1. 文档目的

这份文档用于统一说明当前项目的推送体系，避免后续打开新的 Codex 对话时，对以下问题出现口径不一致：

- 项目里到底有几条推送链路
- 哪些推送属于总推送，哪些属于模块独立推送
- 哪些配置以 `config.yaml` 为默认值
- 哪些配置会被运行时 JSON 覆盖
- 当前云端实际正在使用什么推送设置

本文档是“当前推送体系说明书”，不是历史变更记录。

## 2. 推送体系总览

当前项目共有 4 条真实生效的推送链路：

1. 主摘要定时推送
2. 可转债折价策略提醒
3. 可转债抢权配售独立推送
4. LOF 套利独立推送

统一特点：

- 推送通道固定为企业微信 Webhook
- 文本格式固定为 Markdown
- 推送末尾可追加网页入口链接
- 服务端是唯一主动调度方
- 页面只负责读状态、改配置，不负责真正执行定时推送

## 3. 真实生效链路

### 3.1 总调度入口

总调度在 [start_server.js](/c:/Users/93724/Desktop/Alpha%20monitor/start_server.js) 中组装，核心调度器是：

- [wecom_scheduler.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/scheduler/wecom_scheduler.js)

运行方式：

- 服务器主 tick 每 `60s` 运行一次
- 每次 tick 会检查：
  - 主摘要定时推送是否到点
  - 可转债折价提醒是否需要触发
  - 抢权配售独立推送是否到点
  - LOF 独立推送是否到点
  - LOF 是否有新入池项目需要即时推送

### 3.2 推送发送器

统一发送器在：

- [client.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/wecom/client.js)

职责：

- 校验 `WECOM_WEBHOOK_URL`
- 发送 Markdown 到企业微信
- 若配置了 `PUSH_HTML_URL`，在推送末尾追加网页入口

## 4. 配置真相层级

推送配置有 3 层真相，优先级从低到高如下：

1. `config.yaml` 默认配置
2. `runtime_data/shared/*.json` 运行时覆盖配置
3. 云端 `GET /api/push/*` 返回值

解释：

- `config.yaml` 是“仓库默认值”
- `runtime_data/shared/*.json` 是“某个环境当前保存过的值”
- 云端 API 是“当前线上真实生效值”

因此：

- 要看默认策略，读 [config.yaml](/c:/Users/93724/Desktop/Alpha%20monitor/config.yaml)
- 要看线上现在怎么推，读 `/api/push/config`、`/api/push/cb-rights-issue-config`、`/api/push/lof-arbitrage-config`
- 不要把本地 `runtime_data/shared/*.json` 当成云端真相

## 7. 各推送链路的职责边界

### 7.1 主摘要定时推送

代码入口：

- [main_summary.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/summary/main_summary.js)
- [markdown_style.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/styles/markdown_style.js)

职责：

- 读取多个模块摘要结果
- 按选中的模块拼装统一 Markdown
- 定时发送企业微信摘要

当前支持的摘要模块键：

- `ahab`
- `subscription`
- `cbArb`
- `monitor`
- `dividend`
- `eventArb`

模块含义：

- `ahab`: AH / AB 溢价摘要
- `subscription`: 新股 / 转债申购日历摘要
- `cbArb`: 可转债套利摘要
- `monitor`: 自定义监控摘要
- `dividend`: 分红提醒摘要
- `eventArb`: 事件套利次日摘要

手工触发接口：

- `POST /api/push/wecom`

### 7.2 可转债折价策略提醒

代码入口：

- [event_alert_service.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/alerts/event_alert_service.js)
- [discount_strategy_markdown.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/styles/discount_strategy_markdown.js)

这条链路不属于“主摘要定时推送时间表”，而是独立策略型提醒。

当前策略参数来自 [config.yaml](/c:/Users/93724/Desktop/Alpha%20monitor/config.yaml)：

- `buy_threshold = 2`
- `sell_threshold = 0.5`
- `monitor_interval_minutes = 10`
- `monitor_session_times = 09:30 ~ 14:50`，按 10 分钟一档

它包含 3 类提醒：

- 买入提醒
- 卖出提醒
- 监控名单定时推送

运行规则：

- 默认只在交易时段运行
- 非交易时段除非强制触发，否则直接跳过
- 运行状态记录在折价策略 runtime state 中

手工触发接口：

- `POST /api/push/wecom/event-alerts`

### 7.3 可转债抢权配售独立推送

代码入口：

- [service.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/cb_rights_issue/service.js)
- [cb_rights_issue_markdown.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/styles/cb_rights_issue_markdown.js)

职责：

- 只推送抢权配售模块自己的正式入池项目
- 不并入主摘要

运行规则：

- 默认交易日才推送
- 只按模块自己的时间表推送
- 若监控名单为空则跳过，不记成功

配置接口：

- `GET /api/push/cb-rights-issue-config`
- `POST /api/push/cb-rights-issue-config`

### 7.4 LOF 套利独立推送

代码入口：

- [service.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/lof_arbitrage/service.js)
- [lof_arbitrage_markdown.js](/c:/Users/93724/Desktop/Alpha%20monitor/notification/styles/lof_arbitrage_markdown.js)

职责：

- 推送 LOF 限购池 / 非限购池
- 既支持定时全量推送，也支持新入池即时推送

运行规则：

- 定时推送按自己的 3 个时间点执行
- 即时推送只在“新入池”时触发
- 使用 `seenEntryMap` 去重，避免同一标的重复即时推送

配置接口：

- `GET /api/push/lof-arbitrage-config`
- `POST /api/push/lof-arbitrage-config`

## 10. 本地与云端差异规则

项目已经显式区分本地与云端：

- 若 `PUBLIC_BASE_URL` 是 `127.0.0.1 / localhost / 0.0.0.0`
- 则服务端推送调度会被视为本地运行态并禁用

对应逻辑在 [start_server.js](/c:/Users/93724/Desktop/Alpha%20monitor/start_server.js)：

- `PUSH_SCHEDULER_RUNTIME_ENABLED = scheduler.enabled && !isLoopbackUrl(PUBLIC_BASE_URL)`

因此：

- 本地开发时，即使代码里 `scheduler.enabled = true`，也可能不会真正执行推送
- 云端只要是公网 URL，就会正常启用服务端调度

当前线上实际状态是：

- `schedulerEnabled = true`
- `schedulerDisabledReason = null`

## 11. 建议的运维读法

以后判断“推送是不是正常”的最短路径建议如下：

1. 先看 `/api/health`
   - 重点看 `push_scheduler`
2. 再看 `/api/push/config`
   - 看主摘要时间、选中模块、最近成功/失败
3. 看 `/api/push/cb-rights-issue-config`
   - 看抢权配售最近成功/失败
4. 看 `/api/push/lof-arbitrage-config`
   - 看 LOF 定时推送和即时推送最近成功/失败

如果只是改默认值：

- 改 [config.yaml](/c:/Users/93724/Desktop/Alpha%20monitor/config.yaml)

如果要看线上当前真实值：

- 读线上 `/api/push/*`

如果要清理历史错误认知：

- 不要优先读本地 `runtime_data/shared/*.json`

## 12. 当前结论

当前项目的推送体系已经形成以下稳定结构：

- 主摘要推送：统一总入口，按两个时间点发送
- 可转债折价策略提醒：交易时段内独立策略提醒
- 抢权配售：模块独立定时推送
- LOF 套利：模块独立定时推送 + 新入池即时推送

当前云端真实状态是：

- Webhook 已配置
- 网页入口链接已配置
- 服务端调度已开启
- 主摘要推送正常
- 抢权配售推送正常
- LOF 定时推送正常
- LOF 即时推送正常

如果后续还要继续扩展新的独立推送模块，建议沿用当前已经成型的模式：

1. `config.yaml` 提供默认配置
2. `module_push_config_store` 管理模块配置
3. `module_push_runtime_store` 管理模块运行态
4. 模块自己的 `notification/<module>/service.js` 只负责本模块推送
5. 页面通过独立 `GET/POST /api/push/<module>-config` 管理
